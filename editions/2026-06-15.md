## 1. 🤖 [agent] SWE-Explore：AI 编程 agent 能找对文件，却抓不准该改的那几行

如果你用过 Claude Code、Codex 这类编程 agent，大概都有过这种体验：它能很快翻到出问题的那个文件，但改起来要么不到点子上、要么漏掉关键的几行。上海交大等团队的新基准 SWE-Explore 第一次把这件事量化了——它把「代码定位」从「代码修复」里单拎出来单独评测，给 agent 一段 bug 描述和整个项目，让它返回一份「相关代码段」的排名，再和真正成功的修复实际看过的代码比对。

规模不小：848 个问题、203 个开源项目、10 种语言（Python 547 个为主，也有 Go、JS、Rust）；受测的既有通用 agent（Claude Code、Codex、OpenHands），也有 CoSIL、AutoCodeRover、OrcaLoca 这些专用方案，底座覆盖 GPT-5.4、Gemini 3 Pro、Claude Sonnet 4.6、Kimi K2.6。

结论很扎心：文件级定位大家普遍都行，但落到「行级」，对真正相关代码行的覆盖只有 14–19%。论文给了一句很实在的判断——只要必要核心区露出不到一半，修复大多会失败；而且存在明显的阈值效应：核心区覆盖率从 50% 抬到 75% 这一段，修复成功率会突然跳上去。

这对做 agent 工程的人是个有用的指向标：与其继续卷「能不能找到文件」，不如把力气花在「把对的上下文喂够」。它也解释了为什么很多失败不是模型不会改，而是压根没看全该看的地方。

## 2. 🧠 [model] 微软 Mirage：把「空间记忆」塞进隐空间，长镜头视频世界模型提速 10 倍、省显存 55 倍

视频生成模型有个老毛病：镜头转过去再转回来，刚才的桌子椅子可能就变样了——它「记不住」拐角后面有什么。微软研究院联合多所高校做的 Mirage 想治这个病，思路是给模型一份「空间记忆」。

关键在怎么存这份记忆。以往不少做法用基于像素的 3D 点云，但点云要先渲染、再重新编码回模型，是个「双重瓶颈」，又慢又吃显存。Mirage 直接把图像特征存进模型内部的隐空间（latent space）里当空间记忆，绕过了这道关。它本身是在阿里开源的 Wan2.2 上搭起来的。

省下来的代价很可观：每帧生成最高提速 10.57×，显存最高比同类基于色彩的方法少 55×，而且长距离镜头移动时场景结构和纹理能稳住不漂。基准上，WorldScore 优于 Spatia 和 Wan2.1；在 RealEstate10K 的闭环测试里（镜头绕一圈回到起点，误差会一路累积，是个硬场景）三项指标领先两项。

要泼的冷水也写在脸上：运动物体在分段边界会被刻意滤掉，因为长期几何追不准；安静室内收益大，繁忙场景收益小。但「把空间一致性做进隐空间、还顺手把算力和显存砍下来」这个方向，对世界模型和可交互视频是实打实的一步。

## 3. 🎛️ [post-training] 微软 SkillOpt：不动权重，只「训练」一个 Markdown 技能文件，就把 GPT-5.5 拉高 23 分

提示词工程一直被当成「手艺活」，靠人反复试。微软联合三所中国高校做的 SkillOpt 想把它变成「可训练」的工程：把给模型的指令文档（他们叫 skill）当作一份可训练的外部状态，不动模型权重，只优化那份纯 Markdown 文件。

做法很像把深度学习搬到文本层面：一个独立的「优化器模型」对技能文档提增、删、改，每个改动只有在留出的验证集上表现更好才被接受；还设了编辑预算、拒绝缓冲、按 epoch 巩固这些机制，防止越改越乱。

效果不小。在 GPT-5.5、六个基准的直接对话设置下，平均提升约 23 分；增益最大的是格式要求严、需要工具调用的任务，比如表格编辑。更有意思的是迁移性：在大模型上训出的技能能提升同族的小模型；一个在别处训好的表格技能，原样搬进 Claude Code 就能把它的表现拉到「在 Claude Code 里直接训」的同一水平。实验共覆盖七个目标模型（含 GPT-5.5、Qwen3.5-4B）、六个基准（搜索、表格、文档分析、数学、具身动作）。

这条路线的诱人之处在于便宜且可移植——不需要微调、不碰权重，产出的还是人能读、能改的 Markdown。对手上有一堆 agent、想榨出更多能力又不想动模型的人，这是个很现实的工具。

## 4. 🧠 [model] 1500 美元训出的 10 亿参数 HRM：把「潜空间递归推理」推上下一代架构的牌桌

这次出圈的不是又一个大模型，而是个「小而巧」的反例。Sapient Intelligence 发布的 HRM-Text 只有约 10 亿参数、训练成本约 1500 美元（16 张 H100、不到两天），却把一种叫「潜空间递归推理」（latent recursive reasoning）的架构推到了下一代推理模型讨论的正中央。

它的反常识之处在于：不靠把推理过程「说出来」的显式思维链，而是在模型内部做多轮递归。架构分两层——高层模块 H 慢、管战略，低层模块 L 快、管执行，两个时间尺度在隐空间里反复推演，等想清楚了再出结果。

小归小，分数不寒碜：MATH 56.2、GSM8K 84.5、ARC-Challenge 81.9、DROP 82.2。更能说明热度的是背书——HuggingFace 联合创始人兼 CEO Clem Delangue 亲自转发推荐；图灵奖得主 Yoshua Bengio 作为共同作者的新论文《Generative Recursive Reasoning》（GRAM 框架）也高度复用了 HRM 的骨架，等于把这条「在隐空间里递归地想」的路线往学术主线上又推了一把。

当然 1B 模型的绝对能力有限，几个数学/常识基准也不等于通用推理。但它给行业的提醒很清楚：把算力堆在参数上不是唯一解，「让模型在隐空间里多想几轮」可能是另一条更省的路。

## 5. 💰 [business] KPMG 一份卖 AI 的报告，案例是编的：UBS、NHS 都否认，「二次幻觉」警报拉响

这是个有点讽刺的故事：一家四大之一的 KPMG，发了份名为《Redefining excellence in the age of agentic AI》的报告，劝企业客户大胆上 AI，结果报告本身翻了车——里面列举的多个 AI 落地案例是编造的。

被点名的机构分量不轻：UBS、英国 NHS、瑞士联邦铁路、伦敦交通局，全部出面否认有过报告所述的项目。问题由 AI 检测公司 GPTZero 查出，金融时报独立核实。根因被归为草率的「AI 检索式写作」：引用大多是宽泛转述，缺 URL、缺作者，有些干脆查无原文——GPTZero 给它起了个名字叫「vibe citing」（凭感觉引用）。

比错误本身更值得警惕的是传播方式。GPTZero CEO Edward Tian 提出「二次幻觉」：这类错误因为出自受信任的咨询机构，会被人和 AI 系统反复采信、再加工、再扩散，公信力反而成了错误的加速器。目前 KPMG 已把报告从多个网站撤下。

对正在用 AI 写报告、做尽调的团队，这是一记很实的警钟：模型敢「自信地编」，而越权威的出处，越容易让编出来的东西被当真。

## 6. 📦 [product] 谷歌云 Open Knowledge Format：把散落的组织知识压成带 YAML 头的 Markdown，喂给 agent

AI agent 落地时常卡在一个不性感但要命的环节：它要干活，得先把分散在元数据目录、wiki、代码注释、notebook、甚至工程师脑子里的知识拼起来——比如给某个数据集写 SQL，得四处找口径。谷歌云的 Open Knowledge Format（OKF）想给这件事定个标准。

OKF 的形态很克制：把知识表示成一个目录的 Markdown 文件，每个文件带 YAML frontmatter，只有一个必填字段 type，其余 title、description、resource、tags、时间戳都可选；概念之间用标准 Markdown 链接互连，自然连成一张知识图。它把 Andrej Karpathy 近期带火的「LLM wiki」模式，正式做成了一个可互操作的规范。

卖点是「不绑定」：OKF bundle 任意编辑器能读、GitHub 直接渲染，也不挑云厂商、数据库或 agent 框架。谷歌还附了参考实现——给 BigQuery 数据集做富化的 agent、一个静态 HTML 可视化器，以及 GA4 电商、Stack Overflow、比特币数据集的样例；规范和代码都已在 GitHub 开放。

值得看的不是技术多深，而是方向：当「给 agent 喂上下文」成为刚需，谁能把知识的「装箱格式」标准化，谁就卡住了一个关键接口。OKF 故意做得极简、可移植，正是冲着「被广泛采用」去的。
